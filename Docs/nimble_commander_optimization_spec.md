## 0. Правила выполнения

Обязательны для каждой задачи ниже, без исключений.

1. Выполнять задачи по одной. Один OPT-/LIB-идентификатор = один изолированный коммит. Не смешивать несколько задач в одном диффе.
2. Порядок выполнения: сначала все задачи раздела P0, затем P1, затем P2, затем LIB-*, затем P3 (оппортунистически, можно пропускать по одной с указанием причины).
3. Номера строк в задачах взяты на момент аудита (2026-07-06) и могли сместиться. Прежде чем править — найти актуальное место по приведённой в задаче цитате кода (`grep`/поиск по сигнатуре функции), а не слепо по номеру строки. Если код уже не соответствует описанию (например, кто-то уже это исправил) — пометить задачу как `skipped: already fixed` в итоговом отчёте (см. раздел 8) и перейти к следующей.
4. Перед изменением — прочитать весь модуль/файл целиком, не только указанную функцию, чтобы не сломать вызывающий код.
5. Не менять публичный API модуля (сигнатуры в `include/<Module>/...`) без явного указания в задаче. Если для исправления обязательно требуется менять публичный интерфейс — сначала проверить все точки вызова через `grep -r` по всему `Source/`.
6. После каждой задачи — собрать и прогнать тесты соответствующего модуля (см. `Verification` в каждой задаче). Не переходить к следующей задаче, если тесты не проходят: чинить регресс в рамках текущей задачи, а не откладывать.
7. Общий регресс-прогон: `Scripts/run_all_unit_tests.sh Debug` и, если применимо к изменённому модулю, `Scripts/run_all_integration_tests.sh` — прогонять после завершения всего раздела (P0, P1, P2, LIB) целиком, не после каждой отдельной задачи.
8. Для задач P0 (потенциальные баги памяти) — обязательно собрать через `Scripts/run_all_unit_tests.sh ASAN` соответствующий модуль до и после фикса, чтобы подтвердить, что проблема реальна и что фикс её устраняет.
9. Не расширять область задачи сверх описанного (никакого сопутствующего рефакторинга, переименований, форматирования не относящегося к делу кода).

## 1. Явно вне области работ (не пытаться, не пересматривать)

Эти решения уже приняты по результатам аудита и не подлежат пересмотру в рамках этого документа.

### 1.1 Запрет на переписывание на Go

Не переписывать ни один модуль и ни один компонент на Go — ни целиком, ни частично (включая вынос сетевых VFS-бэкендов FTP/SFTP/WebDAV в отдельный Go-сервис/процесс). Основание: 4 независимые оценки (UI/AppKit-связанность, VFS/сеть, сборка и деплой, риск миграции при ИИ-сопровождении) дали среднюю оценку 1.75/10. Ключевые причины: macOS-специфика (AppKit/CoreText/Carbon/DiskArbitration/FSEvents/XPC/Authorization Services) внедрена на уровне отдельных функций по всей кодовой базе, а не изолирована в одном слое; сборка — через отдельные Xcode-проекты без CMake, нет чистой границы для отдельного Go-бинарника; узкие места, найденные аудитом, упираются в блокирующие syscalls/IPC и алгоритмическую сложность, а не в язык/рантайм.

### 1.2 Библиотеки, которые нельзя переписывать/заменять

Ни при каких обстоятельствах не предлагать и не выполнять переписывание следующих зависимостей из `3rd_Party/`, независимо от объёма кода, который их использует:

- `OpenSSL`, `libcurl`, `libssh2` — реализация TLS/SSH/HTTP-протоколов и криптографии.
- `libarchive`, `lzma` — разбор непроверенных архивов из внешних источников; `lzma` (проект xz) дополнительно связан с supply-chain атакой CVE-2024-3094.
- `pugixml` — разбор XML-ответов WebDAV-сервера (недоверенный сетевой источник).
- `Sparkle` — механизм автообновления (загрузка и установка исполняемого кода).
- `fmt`, `spdlog`, `re2`, `unordered_dense`, `pstld` — структурные зависимости с широким использованием в горячих путях, либо выбраны намеренно ради гарантии (`re2` — защита от ReDoS на пользовательских масках файлов; `pstld` — написана автором Nimble Commander специально под отсутствие Parallel STL в libc++ на macOS).
- `boost`, `lexilla`, `Catch2`, `googletest`, `zstd`, `lz4`, `lzo`, `bz2`, `z` (zlib, за исключением точечной задачи LIB-исключение ниже), `magic_enum`.

Единственные разрешённые точечные правки библиотек — задачи `LIB-001`–`LIB-006` в разделе 6.

## 2. P0 — потенциальные баги (обязательно первыми)

### OPT-001: переполнение стека в `chained_strings` при глубоких деревьях каталогов

- **Файл:** `Source/Base/source/chained_strings.cpp`, функции `node::str_with_pref()` (~строка 137) и `node::to_str_with_pref()` (~строка 155). Константа `max_depth = 128` определена в `Source/Base/include/Base/chained_strings.h:13`.
- **Текущий код (локатор):** обе функции объявляют `const node *nodes[max_depth];` и единственная проверка границ — `assert(nodes_n < static_cast<int>(max_depth));`, которая компилируется в no-op в release-сборке (`NDEBUG`).
- **Проблема:** при цепочке предков длиннее 128 узлов запись `nodes[nodes_n++] = ...` выходит за границы стекового массива. Используется в `Source/Operations/source/Deletion/DeletionJob.cpp`, `AttrsChangingJob.cpp`, `Compression/CompressionJob.cpp` при рекурсивном обходе каталогов — глубина цепочки равна глубине вложенности каталогов, которая на macOS легитимно может превышать 128 уровней.
- **Требуемое изменение:** заменить фиксированный стековый массив на структуру данных без жёсткого верхнего предела (например, `std::vector<const node*>` с зарезервированной начальной ёмкостью, либо small-buffer-оптимизированный контейнер с переходом на кучу при превышении инлайн-ёмкости). Убрать риск переполнения независимо от сборки (Debug/Release). `assert` можно оставить как дополнительную диагностику в Debug, но он не должен быть единственной защитой.
- **Ограничения:** не менять сигнатуры `str_with_pref()`/`to_str_with_pref()` и их возвращаемый тип. Поведение для цепочек ≤128 узлов должно остаться побайтово идентичным текущему.

## 3. P1 — блокировки главного потока (высокий пользовательский эффект)

### OPT-004: отсутствие батчинга в RoutedIO — тысячи последовательных IPC при обходе каталога под правами администратора

- **Файлы:** `Source/RoutedIO/source/RoutedIOInterfaces.cpp` (класс `PosixIOInterfaceRouted`, начиная со `stat()` ~строка 241 и далее — `lstat`, `open`, `chown`, `chflags`, `rmdir`, `unlink`, `rename`, `readlink`, `symlink`, `link`, `chmod`, `killpg`, `trash`); `Source/RoutedIO/source/PrivilegedIOHelper.cpp` (карта обработчиков `g_Handlers`, ~строка 476, 23 обработчика по одному на операцию, без batch-варианта).
- **Реальные потребители:** `Source/VFS/source/Native/Fetching.cpp` (цикл `lstat` по одному на директорийную запись через `RoutedIO::Default`, используется в fallback-пути Admin Mode), `Source/VFS/source/Native/Host.cpp` (8+ точек вызова).
- **Проблема:** каждый вызов проксируется через `xpc_connection_send_message_with_reply_sync` — один IPC round-trip на один POSIX-вызов. Листинг каталога с тысячами файлов под sudo/admin-режимом превращается в тысячи последовательных синхронных IPC-вызовов.
- **Требуемое изменение:**
  1. Добавить в протокол хелпера (`PrivilegedIOHelper.cpp`) новую batch-операцию (например, `HandleStatBatch`), принимающую массив путей и возвращающую массив результатов `stat`/`lstat` за один XPC round-trip.
  2. Добавить соответствующий клиентский метод в `PosixIOInterfaceRouted` (или отдельный API уровня `RoutedIO`), не ломающий существующий одиночный `stat()`/`lstat()`.
  3. Переключить цикл в `Fetching.cpp`, использующий `RoutedIO::Default` для листинга директории, на новый batch-вызов, когда обрабатывается более одной записи.
- **Ограничения:** одиночные `stat`/`lstat`/`open`/... вызовы должны продолжать работать как раньше (обратная совместимость, в том числе при рассинхроне версий хелпера и приложения — если хелпер не поддерживает batch, соответствующий клиентский код должен откатиться на последовательные одиночные вызовы, а не падать). Частичный отказ по одному пути внутри batch-запроса не должен приводить к отказу всего batch — по каждому пути возвращается свой статус.
- **Verification:**
  - [ ] Unit/integration-тест в `Source/RoutedIO/tests` (если каталог тестов существует) или `Source/VFS/tests/Native`, проверяющий, что batch-запрос на N путей возвращает те же результаты, что и N последовательных одиночных запросов, включая случай частичного отказа (несуществующий путь среди прочих).
  - [ ] `xcodebuild -project Source/VFS/VFS.xcodeproj -scheme VFSUT -configuration Debug test` и `-scheme VFSIT` — зелёные.
  - [ ] Ручная/лог-based проверка: количество XPC round-trip на листинг директории из N файлов под Admin Mode должно снизиться с O(N) до O(1) (или до количества batch-чанков, если введено разбиение на пачки).
