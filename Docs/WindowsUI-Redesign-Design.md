# Дизайн-документация: UI в стиле Windows Explorer поверх движка Nimble Commander

> Компаньон к [WindowsUI-Redesign-Plan.md](WindowsUI-Redesign-Plan.md). Тот документ — про то, *как* и *в каком порядке* делать; этот — про то, *что именно* должно получиться: раскладка, компоненты, поведение, палитра, иконки, хоткеи.
> Визуальные референсы (wireframe-макеты) — в отдельном интерактивном артефакте, ссылка в конце документа.

## 1. Принципы дизайна

1. **Узнаваемость важнее точности.** Если пользователь Windows видел элемент в Explorer, он должен опознать его аналог здесь с первого взгляда — по расположению, составу и поведению, не обязательно по пиксель-в-пиксель совпадению.
2. **macOS-платформенные константы не трогаем.** Строка меню — вверху экрана (системная, не в окне), светофор трафик-лайтов, системные жесты (двухпальцевый свайп «назад», Force Click/Quick Look по пробелу), поддержка Retina/several-display setups. Это не «недоделанный Windows», это macOS-приложение с чужим диалектом расположения элементов.
3. **Не факсимиле.** Не копируем проприетарные ассеты Microsoft (Segoe UI, Segoe Fluent Icons, слово «Explorer» как бренд). Ориентируемся на *принципы* (Fluent 2 токены — см. §12), рисуем собственные иконки и используем системную типографику macOS.
4. **Ничего не теряем из возможностей NC.** VFS (архивы/FTP/SFTP/WebDAV/xattr/processes), batch rename, quick search, вкладки, двухпанельный режим, встроенный вьювер и терминал — всё должно быть доступно из нового UI, даже если Explorer-оригинал этого не имеет. Такие «сверхспособности» помечены в тексте как *«наследие NC»*.
5. **Один экран — одна метафора.** Явно решаем конфликтующие метафоры там, где они есть (например, «Corzina» vs «Trash» в §9, «Cut» как визуальное состояние в §6) — вместо того, чтобы оставлять пользователя гадать.

Соответствие текущим классам NC — см. таблицу в §15. Ключевое структурное решение: старый `PanelViewHeader` (путь + сортировка + quick search в одном классе) **разбирается** на три независимых элемента: адресная строка (§4), сортировка/вид (переезжают в командную панель, §5, и в заголовки колонок, §6), quick search (§4.3).

### 2.1. Одна панель по умолчанию, вторая — по требованию

NC уже поддерживает переключение между двухпанельным и однопанельным режимом (`Shift+Cmd+P`, реализовано как схлопывание сплиттера в `MainWindowFilePanelState`). Решение для редизайна: **не удалять и не переизобретать** этот механизм, а изменить лишь *дефолт*:

- при первом запуске приложение открывается в **одной панели**, что визуально соответствует Windows Explorer;
- кнопка **Split View** в командной панели (или тот же хоткей) открывает вторую панель рядом — для тех, кому нужна классическая orthodox-механика NC (синхронизация, drag&drop между панелями, `Cmd+U`/`Opt+Cmd+U`).

Так сохраняется 100% существующей механики панелей, меняется только то, что видно при первом запуске.

## 3. Sidebar / Navigation Pane (новый компонент)

Компонента в NC сегодня нет — вся навигация происходит через модальные Go-To popup’ы и Quick Lists (`Cmd+1`…`Cmd+6`). Хорошая новость: источники данных для этих popup’ов уже существуют и ровно совпадают с тем, что нужно постоянному sidebar.

| Раздел sidebar | Источник данных (уже существует) | Аналог в NC сегодня |
|---|---|---|
| **Избранное** (аналог Quick Access) | `FavoriteLocationsStorage` (`States/FilePanels/Favorites.h`) | Quick List `Cmd+3`, окно `FavoritesWindowController` |
| **Этот Mac** (аналог This PC, тома) | `NativeFSManager` | Quick List `Cmd+4` (Volumes) |
| **Сеть** | `NetworkConnectionsManager` | Quick List `Cmd+5` (Connections), меню `Go > Connect To` |
| **Метки** | `TagsStorage` | Quick List `Cmd+6` (Tags) |
| *(опционально)* **Недавние** | `PanelHistory` | Quick List `Cmd+2` (History) |

Логика агрегации уже написана в `GoToPopupListActionMediator` (`States/FilePanels/Actions/ShowGoToPopup.mm`) — новый sidebar не должен дублировать её, а либо переиспользовать напрямую, либо вынести общий слой агрегации в шаред-класс, из которого одинаково питаются и popup (остаётся для клавиатурных пользователей — `F1`/`F2`), и постоянный sidebar (для мышиных пользователей, в духе Explorer).

**Поведение:**
- реализация — `NSOutlineView` (та же техника, которой пользуется Finder для своего сайдбара — платформенно нативно и производительно);
- секции сворачиваемые (disclosure triangle);
- закреплённые папки в «Избранном» можно перетаскивать для изменения порядка (аналог Quick Access reordering в Explorer);
- правый клик по папке в файловом списке → «Закрепить на панели быстрого доступа» / по элементу sidebar → «Открепить» (использует существующий API `FavoriteLocationsStorage`, только новую точку входа в UI);
- клик по разделу — навигация активной панели, `Cmd+клик` — открытие в новой вкладке (использует существующий `File > New Tab`).

## 4. Breadcrumb Address Bar

Заменяет путевую часть `PanelViewHeader`.

**Устройство:**
- путь показывается сегментами через `›`, каждый сегмент — кликабельная кнопка (переход на этот уровень);
- у каждого сегмента — маленький шеврон-дропдаун со списком «соседей» (папок на этом же уровне) для бокового прыжка без полного перехода;
- клик по пустой области адресной строки (или на иконку в её начале) переключает в режим редактирования как обычного текстового пути — с автодополнением (использует уже существующий движок GoTo-диалога, `Shift+Cmd+G`, только как inline-режим, а не отдельное модальное окно);
- справа в этой же строке — поле поиска.

### 4.1. Поиск в адресной строке — не квик-серч, а вход в Find Files

Важное архитектурное решение: у NC уже есть два разных механизма фильтрации/поиска, и их не нужно путать:

1. **Quick Search** (движок `Source/Panel/include/Panel/QuickSearch.h`) — мгновенная посимвольная фильтрация видимого списка по вводу с клавиатуры. Остаётся как есть по механике, но визуально переезжает из встроенного текстового поля заголовка в **лёгкий плавающий оверлей** над списком (полупрозрачная «таблетка» с текущим запросом и счётчиком совпадений) — это явно нестандартный для Explorer элемент, поэтому визуально обособлен, чтобы не выглядеть как часть адресной строки.
2. **Find Files** (`Cmd+F`, полноценный движок: маска/regex, поиск по содержимому, размер) — по возможностям сильно превосходит стандартный поиск Explorer. Поле поиска в адресной строке — это **точка входа именно сюда**: клик активирует облегчённый инлайн-режим этого диалога (результаты показываются во временной панели, как уже умеет Find Files сегодня — «Reveal all found items in a temporary file panel»), с раскрывающимися «Параметрами поиска» (Тип, Дата изменения, Размер) в духе Explorer.

Никакого нового поискового движка не требуется — только новая раскладка входа в уже существующие два.

## 5. Командная панель (Command Bar)

Заменяет `MainWindowFilePanelsStateToolbarDelegate`.

**Постоянно видимые элементы:**
- `Новый ▾` — подменю «Папка» / «Файл» (использует существующие `File > New Folder` `Shift+Cmd+N` и `File > New File` `Opt+Cmd+N`, оба уже открывают инлайн-переименование сразу после создания — то же самое поведение, что и в Explorer);
- `Сортировка ▾` — переезжает сюда логика из старого дропдауна в заголовке панели;
- `Вид ▾` — переключатель Details/Icons (large/medium/small)/Content, поверх существующих view-mode пресетов (`Ctrl+1`…`Ctrl+5`);
- `Split View` — включатель второй панели (см. §2.1) — это добавка сверх стандартного Explorer, помечена как «наследие NC».

**Контекстные элементы** (появляются при выделении, как в Windows 11 command bar):
- `Вырезать` / `Копировать` / `Вставить` / `Переименовать` / `Общий доступ` / `Удалить` — каждый вызывает уже существующий Action (`Command > Copy To...`, `Rename / Move To...`, `Move to Trash`, и т.д.), кнопка не содержит собственной логики;
- `Общий доступ` — нативная интеграция с `NSSharingServicePicker` (Share Sheet) — то, что у Explorer выглядит как «Share», на macOS реализуется системным API один в один по духу;
- `…` (ещё) — доступ к менее частым действиям: Compress, Calculate Size, Attributes/Properties, Add to Favorites, Batch Rename, Open in Terminal — тем самым командам, что сегодня доступны через меню `Command`.

Правый клик по самой командной панели, как и в Explorer 11, **не даёт** её кастомизировать (сознательное упрощение, не баг).

## 6. Файловый список (режимы просмотра)

| Explorer-режим | Реализация в NC (переиспользуется) | Что меняется |
|---|---|---|
| **Details** (по умолчанию) | `NCPanelListView` (`States/FilePanels/List/PanelListView.h`) | Рестайл строк/заголовков (`PanelListViewTableHeaderView`, `PanelListViewTableHeaderCell.swift`); сортировка по клику на заголовок остаётся, добавляется индикатор направления в стиле Explorer |
| **Icons** (large/medium/small) | `NCPanelBriefView` (Short/Medium режимы) | Рестайл карточек, сетки |
| **Content/Gallery** | `NCPanelGalleryView` | Рестайл под таймлайн-подобную сетку с превью через `VFSIcon` |

**Группировка (Group by).** У Explorer это отдельная возможность поверх сортировки. В `PanelData`/`PanelDataSortMode` группировки сегодня нет — это **небольшое расширение существующей сортировки**, а не новый движок: группа — это визуальная агрегация уже отсортированного списка по значению того же поля, что используется для сортировки (Имя/Дата/Тип/Размер). Реализуется на presentation-уровне (`NCPanelListView`), не в `PanelData`.

**Состояние Cut.** См. §4 Plan-документа (пункт 1) — небольшая новая механика: помеченные «вырезанными» элементы рендерятся полупрозрачными до `Paste`/`Esc`.

**Drag & Drop — сознательное отклонение от буквального Windows-поведения.** Windows: `Ctrl`-перетаскивание = копия, `Shift` = перемещение, без модификатора поведение зависит от того, один диск или разные. На Mac физический `Ctrl` уже занят под вторичный клик, а `Option`-перетаскивание = копия — это въевшаяся привычка любого пользователя macOS (и уже реализовано в NC, `Help.md`: «Dragging items with the mouse while holding the `Opt` key»). Рекомендация: **оставить macOS-конвенцию** (обычное перетаскивание = перемещение/копия по контексту тома, `Option` = копия), но явно подсказывать модификатор в UI (всплывающая подсказка у курсора при перетаскивании, как в Windows) — то есть заимствуем *поведенческую щедрость подсказки*, а не буквальное назначение клавиш.

## 7. Статус-бар

Основа — `PanelViewFooter` (уже принимает готовые данные, чистый рендер-таргет).

- **Слева:** количество объектов в папке; при выделении — количество выделенных и их суммарный размер (сегодня это уже считается для футера NC, просто по-другому оформлено).
- **Справа:** три иконки-переключателя вида (Details/Icons/Content) — быстрый доступ без похода в командную панель, как в Explorer; свободное место на томе (`PanelViewFooterVolumeInfoFetcher`, уже есть).

## 10. Properties (новый диалог поверх существующего движка атрибутов)

Windows Properties — это вкладки General/Security/Details. Предлагаемая адаптация под то, что уже есть в NC:

- **Общие** — иконка, имя, тип, расположение, размер (для папок — по требованию считается через существующий механизм `Calculate Folder Sizes`), даты создания/изменения/доступа (уже показываются в футере панели, здесь просто сведены в одну карточку).
- **Разрешения** — содержимое уже существующего диалога `Command > File Attributes` (`Ctrl+A`): владелец/группа, POSIX-права, флаги, с поддержкой множественного выделения (indeterminate checkboxes) и рекурсивного применения — переносится как есть, без переписывания.
- **Метки** *(опционально)* — назначение тегов через `TagsStorage`.

## 11. Вьювер и терминал

Оба сегодня технически не имеют тулбара: `MainWindowInternalViewerState` возвращает неподключенный `IBOutlet`, `ShellState.windowStateToolbar` явно возвращает `nil`. Это не «сломано», а «ещё не сделано» — оба класса уже реализуют протокол `NCMainWindowState`, у которого ровно для этого есть метод `windowStateToolbar`.

- **Вьювер:** тулбар с переключателем Text/Hex/Preview (`Cmd+1/2/3`, уже есть как хоткеи, добавляются кнопки), поиском; футер (`ViewerFooter`) — рестайл под общий язык статус-баров.
- **Терминал:** минимальный тулбар (например, «показать/скрыть панели» для overlapped-режима, статус синхронизации пути) — на выбор, можно не делать в MVP, терминал может остаться «полноэкранным» инструментом для продвинутых пользователей.

Палитра обоих уже выведена в абстракции (`nc::viewer::Theme`, `nc::term::Settings`) — новая тема (§12) подключается без изменений в `Source/Viewer`/`Source/Term`.

## 12. Тема и визуальный язык

### 12.1. Материалы и поверхности

На Windows это Mica (для окон/титлбаров) и Acrylic (для меню/флайаутов/попапов) — векторы, эмулирующие полупрозрачность и глубину. На macOS для этого **не нужно ничего изобретать**: системный `NSVisualEffectView` с материалами `.sidebar`, `.menu`, `.popover`, `.hudWindow`, `.underWindowBackground` уже даёт практически идентичный эффект нативными средствами. Рекомендация: sidebar → материал `.sidebar` (аналог Mica по духу — приглушённый, полупостоянный фон), контекстные меню/флайауты попапов → `.popover`/`.menu` (аналог Acrylic — светлее, для временных поверхностей). Для справки взяты параметры оригинала: Acrylic использует блюр 30px и пятислойный рецепт (фон, блюр, exclusion-blend, тон, шум) — воспроизводить их не нужно, `NSVisualEffectView` уже решает задачу нативно и быстрее.

### 12.2. Скругления углов

Fluent 2 использует шкалу токенов: `None` = 0, `Small` = 2px (для фигур мельче 32px), `Medium` = 4px (контролы по умолчанию), `Large` = 8px (диалоги/флайауты/меню), `X-Large` = 12px; верхнеуровневые окна WinUI — 8px при 100% DPI. Рекомендация для нашего приложения — **аналогичная по духу, но не идентичная** шкала (macOS уже имеет свои конвенции скругления окон, трогать их не нужно): элементы командной панели/кнопки — 4–6px, карточки/попапы/флайауты sidebar — 8px, крупные диалоги (Properties, Copying) — 10–12px, в духе общей плотности macOS-контролов.

### 12.3. Тени/элевация

Оригинал: 6 токенов тени (2/4/8/16/28/64 px), каждая — композит «резкой» key-тени и мягкой ambient-тени, без параметра spread. Для нашего приложения — 3 уровня достаточно (переусложнение шкалы не оправдано для macOS, где `NSVisualEffectView`/системные тени окон уже решают базовый случай): «на поверхности» (карточки/кнопки, тень почти не видна), «приподнято» (попапы/флайауты sidebar), «модально» (диалоги Properties/Copying — самая выраженная тень).

### 12.4. Цвет

- Поддержка system accent color (`NSColor.controlAccentColor`) — это прямой аналог того, как Windows разворачивает акцентный цвет в 7 оттенков (`Light1-3`/`Dark1-3`/база); на macOS готовое системное API уже покрывает эту задачу, ничего конструировать вручную не требуется.
- Светлая и тёмная темы через существующий механизм `Theme`/`ThemesManager`/`SystemThemeDetector` — см. Plan-документ §2 (архитектура не трогается, меняется JSON-контент).

### 12.5. Типографика

Windows использует Segoe UI Variable (проприетарный шрифт, недоступный и не подлежащий использованию на macOS). Прямого аналога шкалы (Caption 12/16, Body 14/20, Subtitle 20/28, Title 28/36, Title Large 40/52, Display 68/92) достаточно для вдохновения по *плотности и иерархии*, но шрифт — системный на macOS (San Francisco через `NSFont.systemFont`), с аналогичной по духу шкалой размеров (моноширинный — для путей/хэшей/размеров файлов, как и сегодня в NC).

### 12.6. Иконки

Segoe Fluent Icons: сетка 16/20/24/28/48px, монолинейный стиль (одна заливка/обводка), скругления 1–2px, толщина обводки растёт не линейно (~1px на 16–20px, ~1.5px на 24–28px, ~2.5px на 48px), пары Filled/Regular для обозначения активного/выбранного состояния. Это не подлежит копированию (проприетарный набор), но задаёт понятный, воспроизводимый **грамматику** для собственного набора:

- собственная сетка 16/20/24/32/48px (сокращённое подмножество, покрывающее toolbar/sidebar/статус-бар/диалоги);
- монолинейная обводка, скругления 1–2px, лёгкое утолщение обводки на крупных размерах — в остальном рисуется с нуля;
- где нужно состояние «активно» — использовать заливку (filled) вместо обводки (regular), тот же приём, что и в оригинале, но с собственной геометрией;
- иконка приложения (Dock/Finder) — по Apple HIG остаётся полноценной macOS-иконкой (не плоским глифом) — тут платформенная конвенция важнее «Windows-духа».

## 13. Клавиатурные сокращения — новый опциональный пресет

**Важно:** это не замена сочетаний по умолчанию, а второй пресет в уже существующем механизме ремаппинга (`ActionsShortcutsManager`, вкладка `Preferences > Hotkeys`, которая уже поддерживает до 4 сочетаний на действие — см. `Docs/Help.md`). Пользователь выбирает пресет «Windows-style» так же, как сегодня выбирает тему.

| Действие | Windows (эталон, проверено по офиц. справке Microsoft) | Существующий хоткей NC | Решение для пресета |
|---|---|---|---|
| Переименовать | `F2` | Одиночный клик по имени / `Ctrl+F6` | Добавить `F2` как алиас — не конфликтует |
| Новая папка | `Ctrl+Shift+N` | `Shift+Cmd+N` | Уже совпадает по букве `N`, меняется только модификатор под mac-идиому — оставить как есть |
| Удалить → Корзина | `Delete` | `Cmd+Backward Delete` | Для пресета — назначить голый `Delete` на «Move to Trash» (это осознанное отклонение от mac-конвенции ради «прямо как в Windows», нужно явно предупредить об этом в UI пресета, т.к. это самый заметный конфликт с привычками macOS) |
| Вырезать/Копировать/Вставить | `Ctrl+X/C/V` | `Cmd+C/V`, вырезать нет как отдельного состояния (см. §4 Plan-документа, п.1) | `Cmd+X` = «Cut» (пометить + подготовить к Move), `Cmd+V` = Paste (copy или move в зависимости от состояния пометки) |
| Отменить/Повторить | `Ctrl+Z`/`Ctrl+Y` | Нет как файловой операции | Не назначать в MVP (см. Plan-документ §4, п.3 — стретч-цель) |
| Выделить всё / Инверсия | `Ctrl+A` / нет стандартного | `Cmd+A` / `Ctrl+Cmd+A` | Уже совпадает по духу — без изменений |
| Поиск | `Ctrl+F`/`Ctrl+E` | `Cmd+F` (Find Files) | Уже совпадает по букве `F` — без изменений |
| Адресная строка (фокус) | `Alt+D`/`Ctrl+L` | Нет отдельного (GoTo — `Shift+Cmd+G`) | Добавить `Cmd+L` на фокус новой адресной строки (см. §4) |
| Виды (Details/Icons/…) | `Ctrl+Shift+1…8` | `Ctrl+1…5` (Short/Medium/Full/Wide/Gallery) | Оставить существующие — семантически те же виды, просто разное число пресетов |
| Свойства | `Alt+Enter` | `Ctrl+A` (Attributes) | Для нового диалога Properties (§10) назначить `Cmd+I` (macOS-конвенция «Get Info», более естественна для пользователя, чем буквальный `Alt+Enter`, который на Mac менее ожидаем) — сознательный компромисс между «как в Windows» и «не ломать macOS-рефлексы» |
| Новая вкладка/закрыть вкладку | `Ctrl+T`/`Ctrl+W` | `Cmd+T`/`Cmd+W` | Уже совпадает |
| Обновить | `F5`/`Ctrl+R` | `Cmd+R` | Уже совпадает по духу |
| Вверх на уровень | `Alt+Up`/`Backspace` | `Cmd+Up`/`Backward Delete` | Уже совпадает по духу |
| Перетаскивание: копия/перемещение/ярлык | `Ctrl`/`Shift`/`Alt`-drag | `Opt`-drag = копия | См. §6 — сознательно не копируем буквально модификаторы |

Полная сверка проведена по официальной справке Microsoft «Keyboard shortcuts in Windows» и нескольким независимым источникам; отдельно зафиксировано, что «Ctrl+D для удаления» в документации Microsoft есть, но на практике ненадёжен — в наш пресет не включается.

## 14. Дизайн-принцип отклонений от Windows

Каждое сознательное расхождение с буквальным Windows-поведением в этом документе зафиксировано явно (а не тихо забыто), чтобы решения не терялись при разработке:

1. Двухпанельный режим остаётся, просто не включен по умолчанию (§2.1) — «наследие NC».
2. Drag&drop модификаторы — macOS-конвенция, не Windows-буквальная (§6).
3. `Delete` → Trash только в опциональном Windows-пресете хоткеев, не в дефолтной раскладке (§13).
4. Properties открывается через `Cmd+I`, а не `Alt+Enter`, даже в Windows-пресете (§13).
5. Trash остаётся нативным macOS-механизмом, а не собственной «Корзиной» с undelete (Plan-документ §4, п.4).
6. Undo файловых операций — не в MVP (Plan-документ §4, п.3).

## 16. Дорожная карта визуальных итераций после MVP

Не входит в фазы Plan-документа, но логично продолжает направление:

- **Home-страница** (аналог Explorer Home): сетка закреплённых папок + список недавних файлов — данные уже частично доступны через `PanelHistory` и Favorites, требует небольшой новой агрегации, но не нового движка.
- **Ribbon как альтернативный режим командной панели** — для пользователей, скучающих по Windows 10; не приоритет, так как современный Windows 11 сам от ribbon отказался в пользу command bar.
- **Undo последней операции** (см. Plan-документ §4, п.3) — после стабилизации визуального слоя.
- **Полноценный Gallery-таймлайн для фото** поверх существующего Gallery view mode + thumbnails `VFSIcon`.
